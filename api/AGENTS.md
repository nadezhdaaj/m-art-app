# AGENTS.md

## Назначение

Этот файл задает правила для AI-агентов и разработчиков, которые работают над бэкендом проекта `MART`.

Цель: развивать backend под мобильный/игровой клиент на `Unity`, используя текущий стек проекта и не приносить в код неподходящие для этого проекта веб-паттерны.

## Текущий стек

- Runtime: `Bun`
- HTTP framework: `Elysia`
- ORM: `Prisma v7`
- Database: `PostgreSQL`
- API schema generation: `prismabox`
- API documentation: `@elysiajs/openapi`

## Базовые принципы

- Писать backend только в рамках стека `Elysia + Bun + Prisma v7`.
- Сохранять совместимость с уже существующей структурой проекта: `src/`, `src/lib/`, `prisma/`, `src/generated/`.
- Учитывать, что основной клиент этого backend-а это `Unity` приложение, а не браузерный SPA/SSR-клиент.
- Делать API удобным для machine-to-machine/mobile game клиента: предсказуемые JSON-ответы, явные коды статусов, стабильные DTO.

## Правила по аутентификации

- Не использовать `cookie-based auth` как основной способ авторизации.
- Не строить архитектуру вокруг `session cookies`, `httpOnly cookie login`, server-side web sessions и CSRF-ориентированных веб-потоков.
- Основной подход к auth: `Bearer token` в заголовке `Authorization`.
- Предпочтительный вариант: `access token` + `refresh token`.
- Refresh flow должен быть пригоден для Unity-клиента, где токены хранятся и обновляются явно на стороне приложения.
- Все auth-эндпоинты проектировать как JSON API, без редиректов и без зависимости от браузерного поведения.
- Если нужен social login или внешний identity provider, сервер должен принимать и валидировать внешние токены/claims, а затем выдавать собственные bearer tokens для API.

## API conventions

- Все новые эндпоинты должны быть явно типизированы через схемы Elysia.
- По возможности переиспользовать схемы, сгенерированные из Prisma через `prismabox`.
- Для request/response моделей не дублировать вручную то, что уже может быть получено из `prismabox`, если это не ухудшает читаемость.
- Ответы ошибок делать единообразными: machine-readable code, message, optional details.
- Не возвращать HTML.
- Не завязывать API на form submissions, browser redirects и multi-page app flows.

## Prisma guidelines

- Использовать только `Prisma v7` APIs и текущую конфигурацию проекта.
- Prisma client считается сгенерированным артефактом в `src/generated/prisma`.
- Не редактировать вручную файлы внутри `src/generated/`.
- Изменения схемы данных вносить через `prisma/schema.prisma` и миграции.
- При изменении моделей помнить, что проект использует также `prismabox`.
- Предпочитать явные select/include и не тянуть лишние поля из базы без необходимости.
- Raw SQL допустим только когда он действительно оправдан производительностью или отсутствием ORM-возможностей, и это должно быть явно объяснено в коде/комментарии.

## Работа с generated code

- Папки `src/generated/prisma` и `src/generated/prismabox` считаются generated code.
- Агент не должен вручную править generated files, кроме крайних случаев, когда пользователь просит об этом напрямую.
- Если схема изменилась, агент должен менять источник генерации, а не результат генерации вручную.

## Архитектурные ориентиры для MART

- Проект музейный и ориентирован на мобильный/интерактивный опыт.
- Бэкенд должен хорошо поддерживать:
  - профили пользователей;
  - прогресс;
  - достижения;
  - избранное;
  - контент музея: залы, экспонаты, описания, медиа;
  - игровые механики;
  - персонализированные подборки и маршруты;
  - интеграции с виртуальным гидом и AR-сценариями.
- Любые решения должны учитывать, что клиент может работать в сетях с нестабильным соединением.
- Поэтому API лучше проектировать идемпотентным там, где это возможно, а ответы делать простыми для кеширования и повторного запроса.

## Что не стоит приносить в проект

- Cookie/session-first auth
- SSR-first подход
- HTML templating на сервере
- GraphQL по умолчанию без явной причины
- Слишком веб-ориентированные паттерны, завязанные на браузер
- Нерелевантные абстракции ради "enterprise style", если текущую задачу можно решить проще в рамках Elysia

## Предпочтения по структуре кода

- Маленькие и читаемые модули лучше, чем один большой файл.
- Имена сущностей и эндпоинтов должны отражать домен музея и Unity-клиента.
- Бизнес-правила должны быть отделены от transport-layer логики.
- Временные заглушки допустимы, но их нужно явно помечать.
- Если агент добавляет новую подсистему, он должен предложить минимальную, но расширяемую структуру.

## Правила для изменений

- Перед добавлением новой зависимости проверить, решает ли задачу уже текущий стек.
- Если можно решить задачу средствами `Elysia`, `Bun`, `Prisma`, выбирать их.
- Если предлагается auth-решение, оно должно быть совместимо с Unity-клиентом и bearer token flow.
- Если предлагается схема БД, учитывать будущие игровые и контентные сценарии, а не только текущий экран.
- Если меняется `prisma/schema.prisma`, нужно помнить о миграциях и регенерации клиента.
- Если меняется структура базы данных, состав таблиц, связи, enum-ы или значимые поля в `prisma/schema.prisma`, нужно синхронно обновлять `prisma/DATABASE_SCHEMA.md`.
- `prisma/DATABASE_SCHEMA.md` считается **ОБЯЗАТЕЛЬНОЙ** сопровождающей документацией к схеме БД и должен оставаться актуальным после любых изменений моделей Prisma.

## Definition of done для backend-задач

Изменение считается качественным, если:

- оно использует `Elysia + Bun + Prisma v7`;
- не вводит cookie-centric auth;
- совместимо с Unity-клиентом;
- типы и валидация определены явно;
- структура кода остается расширяемой;
- generated code не редактируется вручную без необходимости;
- изменения соответствуют домену проекта `MART`.
