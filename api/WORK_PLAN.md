# План работ по backend

## Цель

Построить серверную часть `M'ART`, которая хранит доменные данные приложения, выдает API для Unity-клиента и поддерживает ключевые продуктовые модули: кастомную авторизацию, прогресс, викторины, галерею, рисунки, новости и ИИ-ассистента.

## Этап 4. Модуль викторин

### Архитектурное решение

- Не вводить отдельную сущность `UserQuizProgress`
- Использовать `QuizAttempt` как единую модель для:
  - текущего состояния прохождения
  - завершенного результата
  - повторных попыток
- Хранить в `QuizAttempt` пошаговый state:
  - `status`
  - `currentQuestionIndex`
  - `correctAnswers`
  - `questionOrderIds`
  - `answeredQuestionIds`
  - `finishedAt`
- Историю прохождений строить по набору `QuizAttempt`, а не по отдельной таблице текущего прогресса
- Если нужно новое прохождение того же квиза:
  - предыдущую активную попытку переводить в `ABANDONED` или `COMPLETED`
  - создавать новую запись `QuizAttempt`

### Задачи

- Создать модели викторин, вопросов, ответов и результатов
- Добавить типы викторин:
  - обычная
  - да/нет
- Реализовать хранение интересных фактов
- Реализовать привязку или случайную выдачу картины из музейного пула
- Сделать API списка викторин
- Сделать API получения одной викторины
- Сделать API отправки результата прохождения
- Связать завершение викторины с системой прогресса

### Результат этапа

- Unity-клиент получает полноценный серверный модуль викторин вместо локальных данных

## Ближайший практический шаг

Первым делом стоит утвердить контракт кастомной авторизации и сразу расширить `Prisma schema` прикладными доменными сущностями пользователя и прогресса. Это позволит безболезненно убрать Firebase из клиентского сценария и открыть путь к викторинам, избранному, рисункам и достижениям без повторной переделки базы.
