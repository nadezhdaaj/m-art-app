# План перехода Unity-авторизации с Firebase на backend

## Цель

Полностью убрать `Firebase Auth` из Unity-части проекта и перевести авторизацию, профиль и связанные пользовательские действия на backend из `elysia-app`.

Этот план является рабочим документом для агентов. Он описывает:

- какие решения уже зафиксированы;
- что именно надо менять в backend;
- что именно надо менять в Unity;
- в каком порядке вычищать Firebase подчистую.

## Зафиксированные решения

### 1. Контракты

- Контракты уже зафиксированы в корне Unity-проекта в виде `OpenAPI` backend.
- Агенты не переизобретают auth flow, а опираются на текущую backend-спеку.
- Если в `better-auth` есть лишние маршруты относительно Unity-сценария, это допустимо, пока нужные маршруты работают стабильно.

### 2. Механизм авторизации

- Основной механизм авторизации для Unity: `Bearer token`.
- Unity получает токен из backend через `headers`.
- Unity хранит токен локально и подставляет его в `Authorization: Bearer <token>` во все защищенные запросы.
- Cookie-session flow для Unity не используется как основной сценарий.

### 3. Email verification

- Подтверждение почты убирается из клиентского сценария.
- Unity больше не блокирует вход по признаку подтвержденной почты.
- Все ветки UI и кода, связанные с ожиданием подтверждения email, подлежат удалению или замене.

### 4. Аватар

- Смена аватара будет происходить файлом с устройства.
- Unity-часть нужно переделать с отправки `URL` на загрузку файла.
- Backend-профиль уже ближе к этой модели и должен считаться целевым направлением.

## Текущее состояние

### Unity

Сейчас логика завязана на `Firebase`:

- `unity/Assets/Scripts/Firebase/FirebaseManager.cs`
  - инициализация Firebase;
  - login/register;
  - отправка email verification;
  - хранение текущего `FirebaseUser`;
  - смена пароля;
  - смена фото профиля по URL;
  - вход в приложение после проверки `user.IsEmailVerified`.
- `unity/Assets/Scripts/Firebase/AuthUIManager.cs`
  - переключение welcome/login/register/verify-email экранов.
- `unity/Assets/Scripts/Firebase/LobbyManager.cs`
  - чтение профиля из `FirebaseManager.instance.user`;
  - отображение `DisplayName`, `Email`, `PhotoUrl`;
  - sign out через Firebase.
- `unity/Assets/Scenes/Registration.unity`
  - содержит прямые ссылки на `FirebaseManager`, `AuthUIManager`, `LobbyManager`.

### Backend

Сейчас уже есть:

- `better-auth` в `elysia-app/src/modules/auth`;
- `Bearer` plugin;
- auth-модели `user/session/account/verification`;
- автоматическое создание прикладного `profile` и `userProgress`;
- защищенный `profile/me`;
- профильный update c загрузкой аватара файлом.

Это значит, что backend уже является целевой системой авторизации, но Unity-клиент еще не переведен на него.

## Объем миграции

Нужно заменить в Unity следующие Firebase-функции:

- регистрация;
- логин;
- восстановление сессии при старте;
- получение текущего профиля;
- смена пароля;
- загрузка аватара;
- выход из аккаунта.

Нужно удалить следующие Firebase-зависимости поведения:

- проверку `FirebaseApp.CheckAndFixDependenciesAsync`;
- использование `FirebaseAuth`;
- использование `FirebaseUser`;
- email verification flow;
- прямой sign out через Firebase;
- загрузку данных профиля из Firebase user profile.

## Целевое состояние после перехода

После миграции Unity должна работать так:

1. При старте приложение проверяет, сохранен ли токен.
2. Если токен есть, Unity делает запрос в backend на получение текущей сессии или профиля.
3. Если backend подтверждает токен, пользователь считается авторизованным.
4. Если токен невалиден, Unity очищает локальную сессию и показывает экран входа.
5. Все данные пользователя берутся из backend-профиля, а не из `FirebaseUser`.
6. Все защищенные запросы идут с заголовком `Authorization: Bearer <token>`.

## План работ для агентов

## Этап 1. Зафиксировать опору на backend-контракт

Задачи:

- Проверить, какой `OpenAPI` файл лежит в корне Unity-проекта и какие auth/profile endpoints реально используются в Unity-сценарии.
- Зафиксировать для Unity минимальный набор используемых маршрутов:
  - login;
  - register;
  - session or me;
  - signout;
  - change password;
  - update profile/avatar.
- Зафиксировать, из какого response header Unity забирает bearer token.
- Зафиксировать единый способ обработки `401`.

Результат:

- У команды есть один согласованный backend-контракт для Unity.
- Агенты не спорят о форме auth flow и не плодят параллельные реализации.

## Этап 2. Вынести клиент авторизации в отдельный backend-слой Unity

Задачи:

- Создать в Unity новый слой работы с backend вместо прямого использования Firebase.
- Разделить ответственность минимум на такие части:
  - auth API client;
  - session/token storage;
  - профильный API client;
  - модели запросов и ответов.
- Перестать использовать `FirebaseManager` как источник истины для пользователя.
- Подготовить единый метод, который автоматически подставляет `Authorization` header.

Рекомендуемая структура:

- `Assets/Scripts/Auth/` или `Assets/Scripts/Backend/`
- классы наподобие:
  - `AuthApiClient`
  - `ProfileApiClient`
  - `SessionStorage`
  - `AuthSession`
  - `CurrentUserProfile`

Результат:

- Unity получает независимый от Firebase слой доступа к backend.
- Следующие шаги уже можно делать без опоры на Firebase SDK.

## Этап 3. Переписать вход и регистрацию

Задачи:

- Заменить login в `FirebaseManager` на запросы к backend.
- Заменить register в `FirebaseManager` на запросы к backend.
- После успешного login/register забирать токен из `headers`.
- Сохранять токен локально.
- После успешной авторизации запрашивать профиль пользователя с backend.
- Переход в приложение делать на основании валидной backend-сессии, а не `user.IsEmailVerified`.

Что удалить:

- `CheckAndFixDependencies`;
- `InitializeFirebase`;
- `FirebaseAuth.DefaultInstance`;
- `SignInWithEmailAndPasswordAsync`;
- `CreateUserWithEmailAndPasswordAsync`;
- проверки `user.IsEmailVerified`;
- `SendVerificationEmail`;
- экран ожидания подтверждения почты, если он больше не нужен продуктово.

Результат:

- Пользователь может зарегистрироваться и войти вообще без Firebase.

## Этап 4. Переписать восстановление сессии при старте

Задачи:

- На старте Unity читать сохраненный bearer token.
- Если токен есть, делать backend-запрос на восстановление текущего пользователя.
- При успехе загружать данные профиля и пускать пользователя в приложение.
- При `401`:
  - очистить токен;
  - показать экран логина.

Важно:

- Unity больше не должна опираться на автоматически восстановленного `FirebaseUser`.
- Источник истины по текущей сессии только backend плюс локально сохраненный token.

Результат:

- Появляется полноценный auto-login через backend.

## Этап 5. Переписать профиль и lobby-часть

Задачи:

- Переделать `LobbyManager` так, чтобы он читал профиль из backend DTO.
- Убрать обращения к:
  - `FirebaseManager.instance.user.DisplayName`;
  - `FirebaseManager.instance.user.Email`;
  - `FirebaseManager.instance.user.PhotoUrl`.
- Подтягивать профиль через `GET /profile/me` или эквивалентный маршрут.
- Обновить отображение имени, email, аватара и прогресса из backend-данных.

Результат:

- Профиль в Unity больше не зависит от Firebase user profile.

## Этап 6. Переделать смену аватара на upload файла

Задачи:

- Убрать сценарий смены аватара по URL.
- Добавить выбор изображения с устройства.
- Отправлять файл на backend в формате, ожидаемом профильным endpoint.
- После успешной загрузки перезапрашивать `profile/me` и обновлять UI.

Нужно проверить:

- каким плагином или нативным способом Unity выбирает файл на Android;
- какой `multipart/form-data` формат ожидает backend;
- как обрабатывать ограничения по размеру и невалидные типы файлов.

Результат:

- Смена аватара работает через backend и файловую загрузку.

## Этап 7. Переписать смену пароля и выход

Задачи:

- Заменить `UpdatePasswordAsync` на backend endpoint смены пароля.
- Заменить `FirebaseAuth.SignOut()` на локальную очистку токена плюс вызов backend signout route, если он нужен для инвалидирования серверной сессии.
- После выхода:
  - очищать локальную сессию;
  - сбрасывать данные текущего пользователя;
  - возвращать пользователя на `Registration`.

Результат:

- Управление сессией и паролем полностью идет через backend.

## Этап 8. Заменить FirebaseManager под чистую

Задачи:

- Определить, нужен ли класс-замена с тем же жизненным циклом, что был у `FirebaseManager`.
- Если нужен, создать новый класс с нейтральным именем:
  - `AuthManager`
  - `BackendAuthManager`
  - `SessionManager`
- Переназначить все UI-кнопки в `Registration.unity` и других сценах на новый компонент.
- Убедиться, что в сценах не осталось методов, привязанных к `FirebaseManager`.
- Убедиться, что `AuthUIManager` и `LobbyManager` больше не используют `FirebaseManager.instance`.

Результат:

- В рантайме больше не существует зависимости на `FirebaseManager`.

## Этап 9. Вычистить Firebase из Unity-проекта полностью

Этот этап выполнять только после завершения и ручной проверки предыдущих этапов.

### 9.1. Вычистить код

- Удалить папку `unity/Assets/Scripts/Firebase`, если в ней больше нет нужного кода.
- Удалить все `using Firebase`.
- Удалить все обращения к `FirebaseAuth`, `FirebaseUser`, `FirebaseApp`.

### 9.2. Вычистить scene bindings

- Проверить `Registration.unity` и другие сцены на отсутствие ссылок на:
  - `FirebaseManager`;
  - старые Firebase-методы;
  - verify-email UI, если он больше не нужен.
- Проверить кнопки, `OnClick`, serialized references и singleton-зависимости.

### 9.3. Вычистить пакеты и артефакты SDK

- Удалить Firebase package dependencies из Unity-проекта.
- Удалить Firebase DLL, editor dependencies и resolver artifacts, если они больше не используются.
- Проверить:
  - `Assets/Firebase`
  - `Packages/manifest.json`
  - `ProjectSettings/AndroidResolverDependencies.xml`
  - связанные `.csproj`, если они пересоздаются с лишними ссылками

### 9.4. Вычистить документацию

- Обновить `unity/README.md`.
- Убрать инструкции по восстановлению `Firebase Auth`, если Firebase больше не нужен.
- Заменить описание auth-стека на backend + bearer token.

Результат:

- Unity-проект полностью очищен от Firebase Auth и связанных зависимостей.

## Проверка готовности

Переход считается завершенным, если выполняются все условия:

- пользователь может зарегистрироваться без Firebase;
- пользователь может войти без Firebase;
- после перезапуска приложения сессия восстанавливается по токену;
- `profile/me` загружается с backend;
- аватар меняется файлом с устройства;
- пароль меняется через backend;
- выход очищает сессию и возвращает на экран входа;
- в сценах нет привязок к `FirebaseManager`;
- в коде нет импортов `Firebase`;
- Unity проект собирается без Firebase SDK.

## Порядок выполнения для агентов

Рекомендуемый рабочий порядок:

1. Проверка и фиксация backend-контракта из OpenAPI.
2. Создание нового Unity backend auth слоя.
3. Перевод login/register.
4. Перевод восстановления сессии.
5. Перевод profile/lobby.
6. Перевод upload аватара файлом.
7. Перевод change password и signout.
8. Замена `FirebaseManager` во всех сценах.
9. Полная зачистка Firebase SDK, документации и зависимостей.

## Ограничения для агентов

- Не удалять Firebase SDK раньше, чем Unity полностью переведена на backend.
- Не оставлять смешанный источник истины, где часть данных идет из backend, а часть из `FirebaseUser`.
- Не возвращать email verification в клиентский flow.
- Не добавлять новый auth-механизм помимо согласованного `Bearer token`.
- Любые расхождения со спекой сначала приводить к backend-контракту, а не обходить временными хаками в Unity.
